# UC-03: Create a New Module with Custom Schema

**Use Case**: Create a new module with a custom schema for a different document type.

## Prerequisites

- Completed UC-01 and UC-02
- Understanding of JSON Schema
- Clear understanding of the document type you want to model

## Steps

### 1. Design Your Document Type

Identify:
- **Purpose**: What kind of documents will this module contain?
- **Fields**: What metadata is required vs optional?
- **Validation rules**: What constraints apply?

**Example**: Module for scientific instruments

Required fields:
- `type`: "scientific_instrument"
- `id`: Unique identifier
- `name`: Instrument name
- `category`: e.g., "telescope", "microscope", "balance"
- `inventor`: Person or organization
- `year_invented`: Year

Optional fields:
- `description`: Detailed description
- `applications`: Array of use cases
- `technical_specs`: Technical specifications

### 2. Create Module Directory

```bash
cd modules
mkdir scientific_instruments
cd scientific_instruments
mkdir documents
```

### 3. Create Module Index

Create `modules/scientific_instruments/index.json`:

```json
{
  "module_id": "SI-001",
  "name": "Scientific Instruments",
  "description": "Historical and contemporary scientific instruments with technical specifications",
  "version": "0.1.0",
  "status": "active",
  "documents_directory": "documents/",
  "schema_ref": "modules/scientific_instruments/schema/scientific_instrument.schema.json"
}
```

### 4. Create Schema Directory

```bash
cd modules/scientific_instruments
mkdir schema
```

### 5. Create Schema File

Create `modules/scientific_instruments/schema/scientific_instrument.schema.json`:

```json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "$id": "https://github.com/[USERNAME]/MWDR-Showcase/modules/scientific_instruments/schema/scientific_instrument.schema.json",
  "title": "Scientific Instrument",
  "description": "Historical and contemporary scientific instruments",
  "type": "object",
  "additionalProperties": false,
  "required": [
    "type",
    "id",
    "name",
    "category",
    "inventor",
    "year_invented"
  ],
  "properties": {
    "type": {
      "const": "scientific_instrument",
      "description": "Type identifier for scientific instruments"
    },
    "id": {
      "type": "string",
      "description": "Unique identifier",
      "pattern": "^INST-[A-Z]+-\\d{4}$"
    },
    "name": {
      "type": "string",
      "minLength": 1,
      "description": "Instrument name"
    },
    "category": {
      "type": "string",
      "enum": ["telescope", "microscope", "balance", "thermometer", "barometer", "chronometer", "other"],
      "description": "Instrument category"
    },
    "inventor": {
      "type": "string",
      "minLength": 1,
      "description": "Inventor name or organization"
    },
    "year_invented": {
      "type": "integer",
      "minimum": -3000,
      "maximum": 2100,
      "description": "Year of invention"
    },
    "description": {
      "type": "string",
      "description": "Detailed description"
    },
    "applications": {
      "type": "array",
      "description": "Use cases and applications",
      "items": {
        "type": "string",
        "minLength": 1
      }
    },
    "technical_specs": {
      "type": "object",
      "description": "Technical specifications",
      "properties": {
        "precision": {"type": "string"},
        "materials": {"type": "array", "items": {"type": "string"}},
        "dimensions": {"type": "string"}
      }
    },
    "provenance": {
      "type": "object",
      "description": "Source information",
      "required": ["source_type"],
      "properties": {
        "source_type": {
          "type": "string",
          "enum": ["book", "paper", "archive", "website", "other"]
        },
        "source_reference": {
          "type": "string"
        }
      }
    },
    "created_at": {
      "type": "string",
      "description": "Creation date",
      "pattern": "^\\d{4}-\\d{2}-\\d{2}$"
    }
  }
}
```

### 6. Add Schema Routing Rule

Edit `src/validate/schema_routing.json`:

```json
{
  "version": "1.0",
  "rules": [
    {
      "glob": "modules/demo_module/documents/*.json",
      "schema": "shared/schemas/epistemic_documents/epistemic_document.schema.json",
      "comment": "Demo module epistemic documents"
    },
    {
      "glob": "modules/scientific_instruments/documents/*.json",
      "schema": "modules/scientific_instruments/schema/scientific_instrument.schema.json",
      "comment": "Scientific instruments module"
    }
  ]
}
```

### 7. Create Example Document

Create `modules/scientific_instruments/documents/INST-TELESCOPE-1609.json`:

```json
{
  "type": "scientific_instrument",
  "id": "INST-TELESCOPE-1609",
  "name": "Galilean Telescope",
  "category": "telescope",
  "inventor": "Galileo Galilei",
  "year_invented": 1609,
  "description": "Refracting telescope using convex objective and concave eyepiece lens",
  "applications": [
    "Astronomical observation",
    "Military reconnaissance",
    "Navigation"
  ],
  "technical_specs": {
    "precision": "Approximately 30x magnification",
    "materials": ["wood", "glass", "leather"],
    "dimensions": "Approximately 1 meter length"
  },
  "provenance": {
    "source_type": "book",
    "source_reference": "Galilei, G. (1610). Sidereus Nuncius"
  },
  "created_at": "2026-03-03"
}
```

### 8. Validate New Module

```bash
cd /path/to/MWDR-Showcase
python3 src/validate/validate_json.py --path modules/scientific_instruments/
```

**Expected Output**:
```
Validation Summary
==================
total_files_scanned: 1
parsed_ok: 1
syntax_errors: 0
schema_valid: 1
schema_invalid: 0
Result: PASS
```

### 9. Validate Full Repository

```bash
python3 src/validate/validate_json.py --path .
```

Should see `total_files_scanned` increase by 1.

### 10. Commit Changes

```bash
git add modules/scientific_instruments/
git add src/validate/schema_routing.json
git commit -m "Add scientific instruments module with custom schema"
```

## Tips

1. **Start simple**: Begin with minimal required fields, add optional fields later
2. **Reuse patterns**: Look at existing schemas for patterns to follow
3. **Test schema**: Create example documents early to test schema design
4. **ID patterns**: Use consistent, descriptive ID patterns for your module
5. **Documentation**: Document your schema in the schema's description field

## Schema Design Best Practices

1. **Use const for type fields**: This ensures type identification
2. **Define patterns for IDs**: Use regex patterns for consistent ID formats
3. **Use enums for categories**: Restrict values to known sets
4. **Make key fields required**: Identify essential metadata
5. **Add descriptions**: Document each field's purpose
6. **Set additionalProperties: false**: Prevent unexpected fields

## Troubleshooting

### Schema Not Found

**Error**: `Schema file not found for rule`

**Solution**: Check that the schema path in routing config matches the actual file location.

### Validation Fails

**Error**: `schema_invalid: 1`

**Solution**:
1. Check the specific error message
2. Verify document fields match schema requirements
3. Check data types (string vs number)
4. Verify enum values are valid

### Pattern Mismatch

**Error**: `Does not match pattern '^INST-[A-Z]+-\\d{4}$'`

**Solution**: Ensure ID matches the pattern exactly (case-sensitive).
